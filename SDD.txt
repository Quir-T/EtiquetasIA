1. OBJETIVO Y ALCANCE
================================================================================

1.1 Proposito
Desarrollar una API REST que:
- anonimize texto clinico,
- procese anamnesis contra un proveedor NLP configurable,
- persista resultados clinicos y trazabilidad operativa de forma inmutable.

1.2 Alcance definido
- Consumidor: sistema PHP interno (legacy).
- Integracion: HTTP JSON con API Key.
- Volumen objetivo: bajo/medio (<10 req/s).
- Entorno soportado en este documento: Docker Compose local.

  "patient_id": 12345,
  "doctor_id": 678,
  "text": "..."
}

{

  "patient_id": 12345,
  "doctor_id": 678,
  "anonymized_text": "..."
}

  "text": "...",

{

  "hallazgos": [

================================================================================
5. MODELO DE DATOS
================================================================================

5.1 Tabla anamnesis_processing_events (clinica/funcional)
- process_id VARCHAR(36) PK NOT NULL
- patient_id INTEGER NOT NULL (index)
- doctor_id INTEGER NOT NULL (index)
- anonymized_text TEXT NOT NULL
- labels_json JSONB NOT NULL
- status ENUM process_status_enum NOT NULL DEFAULT 'success'
  valores: success, provider_error, validation_error, timeout
- created_at TIMESTAMPTZ NOT NULL DEFAULT now() (index)

5.2 Tabla anamnesis_processing_audit (operativa)
- audit_id VARCHAR(36) PK NOT NULL
- process_id VARCHAR(36) NOT NULL UNIQUE (FK -> events.process_id)
- action VARCHAR(50) NOT NULL
- status VARCHAR(30) NOT NULL
- error_code VARCHAR(50) NULL
- error_message TEXT NULL
- processing_ms INTEGER NOT NULL CHECK (processing_ms >= 0)
- prompt_version VARCHAR(50) NOT NULL
- labels_catalog_version VARCHAR(50) NOT NULL
- provider VARCHAR(30) NOT NULL
- provider_model VARCHAR(80) NULL
- metadata_json JSONB NOT NULL
- created_at TIMESTAMPTZ NOT NULL DEFAULT now() (index)

5.3 Cardinalidad
- events -> audit: 1:1 por restriccion UNIQUE en audit.process_id.

================================================================================
6. INMUTABILIDAD Y AUDITORIA
================================================================================

6.1 Politica
- Persistencia append-only en tablas principales.
- Sin endpoints de update/delete.

6.2 Refuerzo en DB
- Funcion: prevent_row_modification()
- Trigger en events: BEFORE UPDATE OR DELETE -> exception
- Trigger en audit: BEFORE UPDATE OR DELETE -> exception

================================================================================
7. MIGRACIONES (ALEMBIC)
================================================================================

Estado de historial:
- 001_initial_schema

La migracion inicial incluye:
- creacion de ambas tablas,
- FK audit.process_id -> events.process_id,
- UNIQUE en audit.process_id (1:1),
- indices principales,
- triggers de inmutabilidad.

Ejecucion en runtime:
- El entrypoint del contenedor api corre alembic upgrade head con reintentos.
